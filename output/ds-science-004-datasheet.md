## Data dictionary

Ensembl genome annotation is most commonly distributed as **GFF3** (and the related **GTF**) — a
9-column, tab-delimited, one-feature-per-line text format. The columns below are **verified** against
the GFF3 specification (Sequence Ontology). Records are coordinate-based features (gene, mRNA,
transcript, exon, CDS, five_prime_UTR, three_prime_UTR, etc.).

| # | Field | Type | Units | Description | Allowed values / null handling |
|---|---|---|---|---|---|
| 1 | `seqid` | string | — | Landmark/sequence ID defining the coordinate system (e.g. chromosome/scaffold name) | Characters outside `[a-zA-Z0-9.:^*$@!+_?-\|]` must be URL-escaped; undefined = `.` |
| 2 | `source` | string | — | Algorithm/procedure or database that generated the feature (e.g. `ensembl`, `havana`) | Free text; undefined = `.` |
| 3 | `type` | string | — | Feature category | A Sequence Ontology term or SO accession (`SO:NNNNNNN`); must be `sequence_feature` or a child; undefined = `.` |
| 4 | `start` | integer | 1-based bp | Feature start position | Positive integer, always ≤ `end`; undefined = `.` |
| 5 | `end` | integer | 1-based bp | Feature end position | Positive integer, always ≥ `start`; undefined = `.` |
| 6 | `score` | float | — | Numeric quality/score for the feature | Floating-point (E-/P-values recommended); undefined = `.` |
| 7 | `strand` | enum | — | Orientation relative to the landmark | `+` (forward), `-` (reverse), `.` (unstranded), `?` (unknown) |
| 8 | `phase` | enum | codon offset | Reading-frame offset for CDS features | `0`, `1`, or `2` (required for CDS); otherwise `.` |
| 9 | `attributes` | string | — | Semicolon-separated `tag=value` pairs of feature metadata | See attribute sub-table; undefined = `.` |

**Column 9 attribute keys** (GFF3-standard keys verified from spec; Ensembl-specific keys marked):

| Attribute | Verified source | Description | Null handling |
|---|---|---|---|
| `ID` | GFF3 spec (verified) | Feature identifier; required for features with children/multi-line | Absent if not needed |
| `Parent` | GFF3 spec (verified) | Links a child feature to its parent (`part_of`) | Absent for top-level features |
| `Name` | GFF3 spec (verified) | Human-readable display name | Optional |
| `Alias`, `Dbxref`, `Ontology_term`, `Note`, `Target`, `Gap` | GFF3 spec (verified) | Aliases, DB cross-refs, ontology terms, notes, alignment target/gaps | Optional |
| `biotype` | **UNVERIFIED (Ensembl extension)** — observed in Ensembl GFF3 but not confirmed against a live Ensembl GFF3 README in this pass | Feature biotype (e.g. `protein_coding`, `lncRNA`) | Mark as unverified pending README check |
| `gene_id`, `transcript_id`, `version` | **UNVERIFIED (Ensembl/GTF extension)** — typical of Ensembl GTF attributes; not confirmed from a live file here | Stable Ensembl identifiers / versions | Mark as unverified pending README check |

> GTF (GFF2-style) shares columns 1–8 with GFF3 but uses a different column-9 syntax
> (`key "value";` pairs, e.g. `gene_id "ENSG..."; transcript_id "ENST...";`). The specific Ensembl
> attribute set per release is documented in the README accompanying each FTP release directory and
> should be confirmed there before relying on extension keys.

## Datasheet for Datasets

**Motivation.** Reference genome annotations describe where genes, transcripts and other functional
features lie on a reference assembly. They are foundational infrastructure for genomics, variant
interpretation, and biomedical research. Ensembl (EMBL-EBI) produces and openly distributes these
annotations; the dense file formats are a recurring barrier for new reusers, which this datasheet
addresses.

**Composition.** Each record is a genomic feature (gene, transcript/mRNA, exon, CDS, UTRs, etc.)
placed on a landmark sequence with start/end coordinates, strand, type (Sequence Ontology), and a
set of attributes (identifiers, biotype, cross-references). Features form a parent/child hierarchy
(gene → transcript → exon/CDS). The data describe genomic sequence annotations, **not individuals**.

**Collection process.** Annotations are produced by the Ensembl gene-annotation pipelines combining
automated computational prediction with manual curation (e.g. HAVANA for human/mouse) and integration
of external evidence (transcript/protein alignments, external databases). Provenance per feature is
partly captured in the `source` column.

**Preprocessing / cleaning.** Ensembl applies its annotation and QC pipelines; features are mapped to
a specific reference assembly and release. Coordinates are 1-based. This datasheet performs **no**
preprocessing of the data itself.

**Uses.** Suitable for gene/transcript lookup, coordinate conversion, feature overlap analysis,
annotation of variants, and as input to genomics tooling. Not a substitute for clinical
interpretation. Users must pin a specific **assembly + release** for reproducibility.

**Distribution.** Distributed openly by Ensembl via its website and public FTP, in GFF3, GTF, FASTA,
and other formats. Data are available without restriction; code under Apache-2.0. **This datasheet
does not redistribute any data.**

**Maintenance.** Maintained by EMBL-EBI / Ensembl on a regular release cadence (named releases with
versioned assemblies). Older releases remain available in the FTP archive. Consult the Ensembl site
for the current release and deprecation policy.

## Validation script

The script below validates the **structure** of a locally-held GFF3 file the user already has. It
**does not download, mirror, or host** any data — the user supplies a local path. It checks the
9-column layout, coordinate sanity, strand domain, and reports a row count.

```python
#!/usr/bin/env python3
"""Validate the structure of a local Ensembl GFF3 file. Documentation/QA only.
Does NOT download or host any data; operate on a file you already have locally.
Usage: python validate_gff3.py /path/to/annotation.gff3 [--max-rows 1000]
"""
import sys, argparse

EXPECTED_COLS = 9
VALID_STRAND = {"+", "-", ".", "?"}

def main():
    ap = argparse.ArgumentParser()
    ap.add_argument("path", help="local GFF3 file path")
    ap.add_argument("--max-rows", type=int, default=1000,
                    help="cap rows scanned (bounded, ephemeral, local-only)")
    args = ap.parse_args()

    data_rows = errors = 0
    with open(args.path, "r", encoding="utf-8") as fh:
        for n, line in enumerate(fh, 1):
            if data_rows >= args.max_rows:
                break
            if not line.strip() or line.startswith("#"):
                continue  # comments / directives (e.g. ##gff-version 3)
            cols = line.rstrip("\n").split("\t")
            if len(cols) != EXPECTED_COLS:
                print(f"[line {n}] expected {EXPECTED_COLS} cols, got {len(cols)}")
                errors += 1
                continue
            seqid, source, ftype, start, end, score, strand, phase, attrs = cols
            try:
                s, e = int(start), int(end)
                if s < 1 or e < s:
                    print(f"[line {n}] bad coordinates start={start} end={end}")
                    errors += 1
            except ValueError:
                if start != "." or end != ".":
                    print(f"[line {n}] non-integer start/end: {start},{end}")
                    errors += 1
            if strand not in VALID_STRAND:
                print(f"[line {n}] invalid strand {strand!r}")
                errors += 1
            data_rows += 1

    print(f"Scanned {data_rows} feature rows; {errors} structural issue(s).")
    sys.exit(1 if errors else 0)

if __name__ == "__main__":
    main()
```

**Expected on a valid Ensembl GFF3:** every feature row has exactly 9 tab-separated columns;
`start`/`end` are positive integers with `start <= end`; `strand` is one of `+ - . ?`; the file
begins with a `##gff-version 3` directive. Row count varies by species/assembly/release.

## Limitations & gaps

- **Licence snapshot not byte-archived in this pass.** The live Ensembl legal/disclaimer pages
  returned HTTP 403 to the authoring fetcher (mirror redirect to `useast.ensembl.org`). The licence
  *terms* are confirmed (Ensembl data: no restriction; code: Apache-2.0) via the Ensembl legal page
  summary and the authoritative Apache-2.0 text and an Ensembl repo LICENSE file, but a hashed,
  archived copy of the exact legal-page text should be captured at the per-dataset gate.
- **Ensembl-specific column-9 attributes are UNVERIFIED here.** `biotype`, `gene_id`,
  `transcript_id`, `version` are typical of Ensembl GFF3/GTF but were not confirmed against a live
  per-release Ensembl README in this pass. Confirm against the README in the relevant FTP release
  directory before relying on them.
- **No release pinned.** Ensembl annotations are versioned by assembly + release; this datasheet is
  format-level. Reusers must pin a specific release for reproducibility.
- **GTF column-9 syntax** differs from GFF3 and is described only briefly here.
- **No data inspected.** Per Elyos guardrails, no dataset rows were downloaded, sampled, hosted, or
  transformed; field definitions derive from the GFF3 specification and Ensembl documentation, not
  from a fetched data file. No personal/identifying data are present in genome annotation features.
